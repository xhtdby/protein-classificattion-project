## Critique of Findings

### 1. The Macro F1 of 0.53 is mediocre — and the project acknowledges but doesn't address it

The best model (XGBoost, F1=0.531) barely outperforms chance on the minority classes:

| Class | F1 | Support |
|---|---|---|
| Lyase | **0.24** | 600 |
| Isomerase | **0.30** | 411 |
| Ligase | **0.49** | 282 |

The report notes these weaknesses but takes no targeted action. Several things could be tried:
- **Per-class threshold tuning** — the model outputs probabilities, but we always take `argmax`. Adjusting decision thresholds per class (especially lowering the threshold for rare classes) could recover recall at marginal precision cost.
- **Focal loss or cost-sensitive boosting** — `sample_weight="balanced"` is a blunt instrument. XGBoost supports custom objectives; a focal loss that down-weights easy (class 0) examples would be more surgical.
- **Two-stage classification** — a binary "enzyme vs not" classifier (trivial at 95%+ accuracy), followed by a 6-class enzyme subtype classifier trained only on enzymes. This removes the massive class 0 imbalance from the hard problem.

### 2. Only the smallest ESM-2 model was used

The project uses `esm2_t6_8M_UR50D` (8M params, 320-dim). The 650M-parameter `esm2_t33_650M_UR50D` (1280-dim) produces substantially richer representations. On an RTX 3060 with 6 GB VRAM, it's tight but feasible with `batch_size=4` or even `batch_size=1`. The plan document even notes this upgrade path, but it was never attempted. Given the embeddings dominate all other features, upgrading the PLM is likely the single highest-ROI improvement and could push Macro F1 above 0.65.

### 3. No hyperparameter tuning was performed

Both XGBoost and LightGBM use hardcoded default-ish hyperparameters:
- `n_estimators=500, learning_rate=0.05, max_depth=6, subsample=0.8`

The plan specified a grid over `max_depth=[4,6,8]`, `n_estimators=[300,500,800]`, `learning_rate=[0.01,0.05,0.1]`, `subsample=[0.7,0.8,0.9]`. None of this was done. Even a quick `RandomizedSearchCV` with 20 iterations would be cheap to run and could yield 2-5% Macro F1 improvement — especially `max_depth` which controls overfitting on minority classes.

### 4. The ablation finding "ESM-2 + Physicochemical is best" was not acted upon

The ablation study showed ESM-2 + Physicochemical (F1=0.535, MCC=0.588) outperforms ESM-2 alone (F1=0.531, MCC=0.581). Yet the best saved model uses ESM-2 alone, because model selection only looks at the "4 model" results, not ablation results. The best ablation combo wasn't promoted to be the final model. This is a missed opportunity and a logical inconsistency in the pipeline.

### 5. Random Forest baseline is degenerate

Random Forest achieved Macro F1=0.128 and MCC=-0.003, which is effectively **worse than random guessing** on the minority classes. This is a red flag that deserves investigation, not just reporting. Likely cause: RF is predicting class 0 for virtually everything (accuracy 0.815 ≈ the class 0 prior). The `class_weight='balanced'` clearly isn't enough, and the 429-dimensional handcrafted feature space (mostly sparse dipeptide frequencies) is too noisy for a tree ensemble without dimensionality reduction. A proper baseline would apply PCA to the dipeptide features first, or use a stratified subsample.

### 6. SMOTE didn't clearly help — the analysis doesn't explain why

The results show:
| Model | F1 | BA |
|---|---|---|
| XGBoost (weighted) | 0.531 | 0.508 |
| XGBoost+SMOTE | 0.527 | 0.541 |
| LightGBM (balanced) | 0.515 | 0.480 |
| LightGBM+SMOTE | 0.526 | 0.523 |

SMOTE improves balanced accuracy but *hurts* Macro F1 for XGBoost. This is counter-intuitive and the report doesn't discuss why. The likely reason: SMOTE generates synthetic samples in 320-dim ESM-2 space by linear interpolation between nearest neighbours. ESM-2 embeddings are *not* linearly interpolable — a midpoint between two enzyme embeddings is not necessarily a valid enzyme embedding. SMOTE on PLM embeddings is a known antipattern in bioinformatics. The report should acknowledge this.

### 7. Interpretability uses Gini importance, which is unreliable for correlated features

The top feature is `ESM_112` by Gini importance. But ESM-2 embedding dimensions are highly correlated. Gini importance is known to be biased toward high-cardinality / high-variance features and unreliable when features are correlated. The SHAP analysis (which is more principled) confirms `ESM_112` is important but shows a much flatter distribution. The report doesn't discuss this discrepancy or note the limitation of Gini importance on correlated embeddings. Permutation importance (which the code supports but doesn't default to) would be a better choice.

### 8. Confidence calibration looks good but hides a problem

High-confidence predictions (p ≥ 0.80) are 95% accurate — impressive. But **28.2% of all errors are high-confidence errors** (1,518 predictions the model was "sure" about but got wrong). For a challenge scoring system where high-confidence errors cost -1 point, this is dangerous. The report should analyze *which classes* have the most high-confidence errors — likely class 0 predictions that are actually enzymes.

### 9. No statistical significance testing

All comparisons are based on mean ± std over 5 folds, but no paired tests (e.g., paired t-test or Wilcoxon signed-rank on fold-level scores) are performed. With only 5 folds, F1 differences of 0.01-0.02 are likely not statistically significant. The difference between XGBoost (0.531) and XGBoost+SMOTE (0.527) could be noise. The report draws conclusions ("balanced weights significantly improve...") without testing significance.

### 10. The predict_blind pipeline doesn't use the best ablation features

`predict_blind.py` uses ESM-2 features only (320-dim), consistent with the saved model. But if the ablation result (ESM-2 + Physicochemical) were promoted, the blind pipeline would also need to compute physicochemical features. This is another consequence of finding #4 — the best feature combination was identified but not deployed.

### Summary: What's good

- **Data leakage prevention** is solid — scaler fit on train fold only, SMOTE on train fold only, OOF evaluation.
- **ESM-2 embeddings** as the primary representation was the right call.
- **Balanced sample weights** for XGBoost was a crucial fix that delivered +14% F1.
- **Confidence calibration** report is well-structured and the thresholds match the spec.
- **SHAP + Gini + confusion analysis** covers 3 interpretability techniques (more than the required 2).
- **15 figures** generated, covering all required visualisations.

### What needs addressing (in priority order)

1. Try `esm2_t33_650M_UR50D` — likely +10-15% F1
2. Run hyperparameter tuning (even 20-iteration random search)
3. Promote ESM-2 + Physicochemical as the deployed model
4. Investigate the degenerate RF baseline (apply PCA to dipeptides)
5. Add per-class threshold optimization
6. Discuss SMOTE limitations on PLM embeddings
7. Analyze which classes produce high-confidence errors

User: what's fine tuning esm-2 8m going to do

GitHub Copilot: Right now you're using ESM-2 as a **frozen feature extractor** — it was pretrained on millions of protein sequences to predict masked amino acids, not to distinguish enzyme classes. The 650M/8M embeddings are general-purpose representations.

**What fine-tuning changes:**

The gradient flows *back through* the transformer layers during training, so the attention heads learn to pay attention to the specific sequence patterns that distinguish Lyase from Isomerase, rather than patterns useful for general protein structure prediction.

**Why it would help your specific problem:**

The classes that are failing (Lyase F1=0.38, Isomerase F1=0.31) are hard because their distinguishing features are subtle catalytic site motifs — short sequence regions that matter a lot but are diluted when you mean-pool 1022 positions into one 320-dim vector. With fine-tuning, the attention layers can learn to *up-weight* those positions.

**What you'd actually build:**

```
ESM-2 8M (6 transformer layers, unfrozen)
    ↓  mean pool
  [320]
    ↓  Linear + ReLU + Dropout(0.3)
  [128]
    ↓  Linear
  [7]   → softmax → class probabilities
```

Weighted cross-entropy loss (same `class_weight='balanced'` logic). 5-fold CV, early stopping on val F1.

**Realistic improvement:** 0.60 → 0.65–0.70. Not guaranteed, but plausible because you're now optimising directly for the classification signal rather than hoping a general embedding captured it.

**The honest trade-off:** It's another ~3 hours locally and another ~400 lines of PyTorch code. Whether that's worth it depends on how much time you have for the report vs. how much a few F1 points matter to you.
